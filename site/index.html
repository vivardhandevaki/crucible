Determinism for AI-driven development

Nothing leaves the crucible unverified.

Crucible is an opinionated framework for building software end-to-end with AI agents, inside a container of enforcement that never yields.

Humans author the specifications and the executable tests that judge code. Agents author the implementation. And a deterministic state machine — enforced by git, CI, and branch protection — guarantees that no code reaches production without passing through every mandatory verification step.

state machine + gates + harness + console humans review the judges, not the code git is the database
01 — Purpose & thought process

Why Crucible exists

The problem isn't that AI can't write the code. It's that verification effort is conserved, not eliminated.

ELI5

AI can write most code by itself now. So people say: “stop checking it, just trust it.” But someone still has to make sure the code is right. If the human stops looking and nothing takes their place, the code isn't trusted — it's just unchecked. Those are very different things.

Crucible's fix: let the AI write code fast, but it has to pass tests that a person wrote and okayed first. Think of it like a teacher who writes the answer key before the students take the exam. One answer key can grade a thousand exams the exact same way — fast, and fair. The human's job changes from “read every line of code” to “write a really good answer key.”

A crucible is a vessel that withstands extreme heat while the material inside is melted, purified, and tested. In this system, AI agents do fast, high-volume work inside a container of enforcement that never yields. That metaphor is the whole design.

The naive responses both fail:

  • “Stop reviewing PRs, trust the model.” Verification doesn't vanish; it silently becomes absence of verification.
  • “Review everything as before.” Throws away the productivity gain and can't scale to agent-level PR volume.

So Crucible makes one core inversion relative to traditional development:

The inversion The artifact humans review shifts from code to the things that judge code. Line-by-line human review is kept only for a mechanically-identified high-risk slice (~15–20% of PRs) plus a random audit sample. Everything else is verified by CI gates and an adversarial reviewer agent.
Why is judging the judges better than reading the code? Why this?

Three arguments, each independent:

  • Leverage. A human reading a diff checks one change. A human approving an oracle checks every future change that oracle will ever judge — including changes that don't exist yet. Effort spent on the judge compounds; effort spent on the diff does not.
  • It doesn't scale the wrong thing. Agent PR volume grows without bound. Human reading time doesn't. If trust depends on reading, throughput is capped by the slowest human; if trust depends on gates, throughput is capped by CI, which scales horizontally.
  • Humans are worse at the code-reading task than we admit. Line-by-line review reliably catches style and obvious logic errors and reliably misses the things that actually hurt: a missing index that's quietly quadratic, a race condition, a subtly weakened assertion. Those are exactly what property tests, mutation testing, and static analysis catch deterministically.

The counter-argument — “but a human sees things no test can” — is accepted, not dismissed. That's precisely why ~15–20% of PRs (the risky ones) plus a random 10% audit are still read by a human. Review is kept as a scalpel, not thrown away as a blanket.

The failure modes it's designed against

Unverified AI-driven development fails in specific, repeatable ways. Each is a thing Crucible is built to catch mechanically.

The six failure modes, in detail Drill down
  • Vacuous verification. The agent writes both the code and the tests, so the tests describe what the code does, not what it should do. Forty green tests can encode one misunderstanding.
  • Reward-hacking to green. Agents optimize “make the checks pass” and find degenerate paths — weakening assertions, editing fixtures, mocking away the failing component, pinning a different library version, mutating shared state.
  • Blind spots of behavioral testing. Security holes with no functional symptom, license/supply-chain risk in hallucinated dependencies, quietly quadratic algorithms that pass on small fixtures, PII in logs, semantic drift near the task.
  • Architectural erosion. Agents have no taste and no long-term memory of your architecture. Without mechanical boundaries, hundreds of PRs converge on a ball of mud.
  • Process drift. LLMs are unreliable at self-enforcing multi-step processes. “Usually follows the workflow” is exactly what repeatability forbids.
  • Correlated review errors. An AI reviewer that shares the author agent's context inherits its rationalizations — it will bless the same mistakes.
Example: how “40 green tests” can encode one bug Example

Say the requirement is “a refund is the item price times the quantity.” An agent that writes its own test often writes the test from the same idea it used to write the code — so if the idea is wrong, both are wrong together and the test still passes:

the code (has a bug: forgets quantity)
Money refundFor(Item item, int qty) {
  return item.price();            // BUG: should be price × qty
}
the agent's own test (encodes the SAME bug — passes anyway)
// re-derives "expected" with the same wrong idea
Money expected = item.price();          // ← also forgets qty
assertEquals(expected, refundFor(item, 3));   // GREEN ✓ (but wrong!)

The test is green. Add thirty-nine more tests written the same way and you have forty green tests that all agree with one misunderstanding. This is why the test that judges the requirement must be written by the human, up front, and must state the rule independently of the code — see the oracle example in the next section.

The four design requirements, in priority order

01

Correctness & verifiability

Every requirement is judged by a machine-executable oracle. Human trust is placed in oracles and gates, not in reading every diff.

02

Extensibility

Architecture rules are enforced mechanically, not culturally, so hundreds of agent-authored PRs cannot erode structure. Adding a workflow step is “add a required check,” not “rewrite an orchestrator.”

03

Repeatability

The agent harness is versioned code with its own regression suite. Builds are hermetic; toolchain and schema versions are pinned. Same inputs, same results.

04

Reliability

Pre-merge gates plus runtime verification (canary, SLO auto-rollback) plus a ratchet: every escaped defect becomes a new permanent oracle, rubric line, or architecture rule.

02 — The words that matter

A shared vocabulary

Consistent names matter because agents read these documents too. A few terms carry the whole model.

ELI5

The most important word here is oracle. An oracle is just an automatic judge — a little program that says “pass” or “fail” all by itself, no human needed. It could be a test, or a database rule.

The slogan to remember: “a requirement without an oracle is a wish.” If there's no automatic judge for a rule, the AI treats that rule as optional — and when the AI is trying hard to make everything pass, anything optional quietly disappears. So if you really want something, write a judge for it.

Intent vs. spec vs. oracle — the distinction that is the crux Opinionated

These three are constantly conflated, and the whole system depends on keeping them apart:

  • Intent = the human-level goal, in natural language. What you want.
  • Spec = the sharpened, precise requirement (SHALL/MUST + scenarios). What you said.
  • Oracle = the executable judge that decides pass/fail. What actually decides.

Agents are steered by specs but judged only by oracles. Anything in the intent or spec that never became an oracle is, from the agent's perspective, optional. That's why the oracle — not the prose — is the load-bearing artifact.

What an oracle actually looks like

The abstract definition lands better with code. Follow one requirement from English, to a precise spec line, to the executable judge — and see why the judge is written as a property (true for all inputs) rather than a single example.

Example: one requirement → spec → oracle, in code Example

1. Intent (what a human wants, in plain words):

“We should never refund a customer more money than they actually paid.”

2. Spec (the sharpened, normative requirement):

openspec/changes/partial-cancellation/specs/refund/spec.md
REQ-PC-2: The sum of all refunds issued against an order
          SHALL NOT exceed the amount captured for that order.

3. Oracle (the executable judge — a jqwik property test):

/oracles/refund/RefundInvariants.java · human-owned, agents cannot edit
@Property                                  // runs on hundreds of generated cases
void refundsNeverExceedCaptured(
        @ForAll("orders") Order order,        // ANY valid order…
        @ForAll("cancellations") List<Cancel> cancels) {  // …ANY cancellations

    Money refunded = Money.ZERO;
    for (Cancel c : cancels)
        refunded = refunded.add(service.cancel(order, c).refund());

    // the invariant the HUMAN cares about — independent of how refunds are computed
    assertThat(refunded).isLessThanOrEqualTo(order.amountCaptured());
}

Two things make this an oracle and not just a test. First, it states the rule independently of the implementation — it never re-computes the “expected” refund with the code's own formula, so a bug in that formula can't hide (contrast the vacuous test in the previous section). Second, it's a property: the @ForAll generators throw hundreds of orders and discount shapes at it, so it catches whole classes of bug an example like “refund on a $10 item” never would.

The same requirement can carry more than one oracle. For the highest-stakes rules you'd add a second, independent judge at a different layer — e.g. a database constraint:

/oracles/refund/V12__refund_cap.sql · a second, independent oracle
-- even if application code is wrong, the DB refuses to over-refund
ALTER TABLE refund_ledger
  ADD CONSTRAINT refund_within_capture
  CHECK (total_refunded <= amount_captured);

And the traceability linter (a Gauntlet gate) refuses to go green unless every SHALL is mapped:

oracles.md — the requirement → oracle map
| Requirement | Oracle(s)            | What it checks                    |
|-------------|----------------------|-----------------------------------|
| REQ-PC-2    | ORA-PC-2a, ORA-PC-2b | Σ refunds ≤ amount captured        |
In plain words

A normal test says “if I refund this one order, I should get this exact number.” An oracle says “no matter what order or discount you throw at me, the refunds can never add up to more than the customer paid.” The second one is a promise about everything, not an example. That's the whole idea.

03 — Core principles

The constitution

Ten principles. When any design question arises, it is resolved in favor of these.

ELI5

All ten rules come from two simple ideas.

One: don't ask the AI to remember the steps — build the steps into the tools. An AI is like a smart worker who sometimes forgets the checklist. A locked door never forgets. So Crucible uses locked doors, not reminders.

Two: “easy” and “required” are different jobs. The app makes doing it the right way easy. Git and the checks make doing it the wrong way impossible — even for the boss. Delete the nice app and the locked doors are still locked.

Principle 01

The loop lives in infrastructure, never in an agent's head

Agents are stateless workers invoked at specific states with one job each. The orchestrator is deterministic code — scripts and CI — debuggable with cat and git log.

Principle 02

Hand-offs are artifacts, not messages

Each stage consumes files and produces files. “Done” means the output artifact exists and validates — never the agent's self-report.

Principle 03

Convenience and enforcement are separate layers

The CLI and Console make the workflow easy; branch protection, CODEOWNERS, and required checks make it impossible not to follow. Deleting the CLI leaves the workflow enforced — just tedious.

Principle 04

Direction emerges from preconditions

No component knows the whole flow. Each step refuses to run unless the previous step's artifact exists and validates. Adding a step = adding a precondition.

Principle 05

Agents are judged only by oracles

Anything that never became an oracle is optional from the agent's view — and under optimization pressure, optional means gone. Hence: a requirement without an oracle is a wish.

Principle 06

Humans own specs, oracles, and the harness; agents own implementation

The protected paths (/specs, /oracles, /harness, /ci, /skills) are CODEOWNERS-guarded and unwritable from implementation branches.

Principle 07

The test bench is deterministic AND tamper-proof

Hermetic, reproducible, sandboxed, destroyed after — and the thing under test, which is also an actor with a shell, cannot modify the bench.

Principle 08

Independent verification: the reviewer never shares the author's context

Different agent, different inputs (spec + diff + rubric; never the implementer's reasoning), concrete rubric, machine-parseable verdict.

Principle 09

Small diffs, always

≤400 changed lines per PR (configurable). Larger work is decomposed into a PR sequence up front. Small diffs make every gate more discriminating and human review tractable.

Principle 10

The ratchet: escaped defects become permanent gates

Every postmortem closes with “which new oracle / rubric line / rule prevents recurrence” — and that item is a commit, not a document.

Example: “judged only by oracles” — how an agent reward-hacks, and what stops it Example

An agent under pressure to make a check pass will do the laziest thing that turns the light green. A classic move is to quietly weaken its own test:

the agent weakens an assertion to escape a failing check
// Requirement: the ledger must always balance to exactly zero
- assertThat(ledger.balance()).isEqualTo(Money.ZERO);
+ assertThat(ledger.balance()).isNotNull();   // ← "technically passes" 🙈

Three separate parts of Crucible make this fail anyway:

  • The real check is an oracle the agent can't touch. The balancing rule lives in /oracles (human-owned). The agent can weaken a test it wrote, but not the judge that actually gates the merge.
  • Mutation testing catches weak assertions. A test that only checks isNotNull() lets almost every injected mutant survive → the mutation gate goes red.
  • The reviewer agent flags it. A separate agent, looking only at the diff and the spec, sees an assertion that got weaker in a change that claims to add behavior — and raises it.

The lesson of principle 5: don't rely on the agent's good intentions. Make the thing it can't reach be the thing that decides.

Why cap diffs at ~400 lines? Isn't that just bureaucracy? Why this?

Small diffs aren't a style preference — they make every other part of the system sharper:

  • Gates get more discriminating. When mutation testing or the reviewer flags a 40-line PR, the problem is right there. In a 2,000-line PR, a red gate points at a haystack, and the temptation to override it grows.
  • Human review becomes tractable. The risk-routed 15–20% has to be actually readable in minutes. That only works if diffs are small.
  • Rollback is cheap. A small merged change that misbehaves in canary can be reverted with near-zero blast radius. Big changes entangle.
  • It forces decomposition up front. “This needs 900 lines” is really “this is several changes.” Splitting them into a planned sequence surfaces the seams — and the risky one gets routed to a human on its own.

The cap is per-work-order configurable, so genuinely atomic large changes have an escape hatch — but it's a deliberate, logged decision, not the default.

The one operating rule No code without a work order; no work order without approved oracles — including hotfixes, including your own “quick” ideas. The loop is kept fast precisely so there is never a reason to go around it.
04 — Architecture & design

How it's built

Four layers, a state machine with a gatekeeper on every edge, and a verification stack where each gate catches something the others miss.

ELI5

Picture a factory with four floors, bottom to top:

  • Floor 1 — the building itself. The walls, the safety rules, the machines. Built once.
  • Floor 2 — the design room. Humans decide what to make and exactly how it will be inspected.
  • Floor 3 — the assembly line. The AI builds the thing, then it goes through a long row of inspection machines.
  • Floor 4 — the loading dock. After the product ships, sensors watch it. If it acts up, it gets pulled back automatically.

And one rule for the whole factory: when a station is done, it puts a real part on the conveyor belt for the next station. Nobody just shouts “it's done!” — you can always point at the part.

The four layers

Layer 4 — Runtime verification per deploy

Canary deploys, SLO monitoring, auto-rollback, postmortem. The ratchet feeds new oracles and rules back down into Layer 1.

Layer 3 — Execution loop per work order

Task packaging → ephemeral sandbox → implementer agent → PR → the Gauntlet → reviewer agent → deterministic routing → auto-merge or human review.

Layer 2 — Spec layer per feature · human-owned

Proposal → specs → design → oracles → tasks. Plus the oracle map and the oracle implementations on a protected path, sealed by human approval commits.

Layer 1 — Foundation built once, evolved deliberately

Repo structure, CODEOWNERS, branch protection, pinned toolchain image, Gauntlet CI definitions, the harness (CLI, CLAUDE.md, skills, prompts, rubric, sandbox runner), and the pipeline eval suite.

The Crucible Console sits beside all four layers as a stateless view and remote control — it reads state from git + the GitHub API and triggers actions that already exist in the tooling.

Who owns what

The cleanest way to hold the whole system in your head: three actors, each with a fixed, non-overlapping job. Nobody reaches into anyone else's lane.

Humans own
  • Specs (the requirements)
  • Oracles (the judges)
  • The harness & the gates themselves
  • Risky-PR & sample review
  • Escalation decisions
Agents own
  • Implementation code
  • Their own unit tests
  • Drafting specs/oracles for a human to approve
  • Writing an escalation when stuck
Machinery owns
  • Advancing the state machine
  • Running the Gauntlet
  • Routing PRs (no model call)
  • Merge, canary, rollback
  • Opening the PR (the runner, not the agent)

The boundary is enforced, not suggested: the human lanes map to CODEOWNERS-protected paths that implementation branches literally cannot write to.

The state machine

Each feature is a work order that moves through explicit states. State is stored durably in workorder.yaml, advanced only by the CLI or humans, and cross-checked by CI.

DRAFT_SPEC → SPEC_APPROVED → ORACLES_AUTHORED → ORACLES_APPROVED → PACKAGED → IMPLEMENTING → PR_OPEN → GATES_GREEN → AI_REVIEWED → ROUTED{AUTO_MERGE | HUMAN_REVIEW} → MERGED → CANARY → DONE → ARCHIVED Failure edges: IMPLEMENTING → ESCALATED → (human spec/oracle fix) → back GATES_RED → IMPLEMENTING (failure output becomes new input) CANARY_ROLLBACK → postmortem → ratchet commit → new work order

Example: the work order — the file the whole state machine turns on Example

The state isn't in a database or an app — it's a plain YAML file in the repo, advanced only by the CLI or a human and cross-checked by CI. You can read the entire status of a feature with cat:

/workorders/OMS-142-partial-cancellation/workorder.yaml
id: OMS-142
state: ORACLES_APPROVED           # the current position in the state machine
change: partial-cancellation     # the OpenSpec change folder (the "content")
scope:
  modules: [order/cancel, refund]  # the agent may ONLY touch these paths
constraints:
  max_diff_lines: 400            # the small-diffs cap, per work order
  iteration_budget: 6            # stop-and-escalate instead of thrashing forever
oracles: [ORA-PC-1a, ORA-PC-2a, ORA-PC-4a, ORA-PC-5a]  # the judges it must pass

Everything follows from this file. scope.modules is what the work-order-legitimacy gate checks touched paths against. state is what every precondition reads. Because it's just a committed file, git log is a complete, tamper-evident history of how the feature moved.

The rest of the hand-off directory carries the other artifacts — each stage produces a file, never a message:

/workorders/OMS-142-partial-cancellation/
  workorder.yaml       # state, scope, constraints, oracle refs
  escalation.md        # exists ONLY if the agent got stuck
  review-verdict.json  # the reviewer agent's machine-readable verdict
  runlog/              # full sandbox transcripts, per attempt (audit trail)

The Gauntlet — what each gate uniquely catches

Each gate exists because it catches a failure mode the others miss. None is optional; together they are what replaces routine line-by-line human review.

ELI5

The cleverest gate is mutation testing. Normal tests check the code. Mutation testing checks the tests.

Here's the trick: it secretly breaks the code on purpose — changes a plus to a minus, flips an “if” — and then watches. If your tests are any good, at least one should now fail and yell. If you break the code and every test still passes, then the tests were fake all along. This is the main thing stopping an AI from writing tests that pass no matter what.

Example: how mutation testing exposes a fake test Example

Suppose the code correctly blocks cancelling a shipped item:

the code under test
if (item.isShipped()) reject();   // shipped items can't be cancelled

Mutation testing (PIT) makes tiny, deliberate breakages — mutants — and reruns the tests:

PIT flips the condition
if (!item.isShipped()) reject();  // MUTANT: now rejects the WRONG items
  • Strong suite — a test that actually cancels a shipped item and expects rejection now fails. The mutant is “killed.” Good.
  • Vacuous suite — if every test still passes with the logic inverted, the mutant “survived.” Your tests never really checked this rule. The mutation gate goes red and the PR cannot merge.

This is why coverage numbers don't impress Crucible: you can execute a line without checking what it does. Mutation score can't be faked that way — surviving mutants are proof the assertions are hollow. The threshold is ratcheted upward over time (75% → 85%).

Where the code lives (Model B)

This repo is the framework — reusable machinery, developed conventionally. The software systems you build with it live in separate consumer repos, scaffolded and governed by crucible init. Distribution is via an npm CLI, reusable GitHub workflows, and a published toolchain image.

Why framework-and-consumers instead of one self-hosting monorepo? Opinionated

An earlier design (“Model A”) put the machinery and one target system in a single repo that governed itself. The current design (“Model B”, ADR 0001) splits them:

  • The framework repo holds the reusable core, gates, harness, schema, and the Console. It's governed conventionally — the owner reviews all PRs.
  • Consumer repos are your actual software systems. Each gets the governed structure — CODEOWNERS, the Gauntlet CI, the oracle-driven schema, CLAUDE.md — emitted by crucible init.

The win: one framework governs many systems, each isolated, each reproducibly scaffolded — rather than entangling the tool with any one product it builds. The Console is a co-primary deliverable (a thin view over a working core), not a last-minute add-on.

05 — How it works, behind the glass

What actually happens when you press a button

The Console is a pane of glass. Behind it, every action is a real git commit, a real GitHub PR, and a real CI run — the same things you could do by hand. There is no hidden application state anywhere.

ELI5

Think of a self-checkout machine at a shop. The screen is friendly, but it isn't keeping your money — the till and the bank do that. The screen just presses the same buttons a cashier would.

Crucible works the same way. The pretty app is the screen. The real record — who approved what, which tests passed, what state your feature is in — lives in git and GitHub, the same place your code already lives. If the app vanished this second, every fact would still be sitting there, untouched. Nothing important is ever inside the app.

The six things going on under the hood

You don't need to think about any of these to use Crucible — but this is the machinery that makes the guarantee real rather than merely promised.

1 · Git is the database

There is no server holding “the truth.” A feature's state, its approvals, the reviewer's verdict, the full sandbox transcript — each is a committed file or a GitHub object. That's why the app can be stateless: it has nothing to lose.

2 · The Console only reads & delegates

It reads state with git + the GitHub API, and the only way it ever writes is by shelling out to the same crucible CLI or opening a PR — never a private back-channel. Every button shows its exact terminal equivalent.

3 · The CLI advances, CI enforces

crucible moves a work order forward in workorder.yaml and checks preconditions — but it does not enforce. It's a convenience layer. Skip it and you simply hit the wall in the next two rows.

4 · The platform is the wall

GitHub branch protection makes the Gauntlet a set of required checks. No one — not even the repo owner — can merge around them. This is why the guarantee holds against a determined human, not just a well-behaved one.

5 · The sandbox is sealed & detached

crucible run spins a fresh, pinned container with the network off and /oracles, /specs, /ci mounted read-only. It runs detached — killing the Console never aborts a run in flight.

6 · Everything runs as CI, unprompted

The Gauntlet, the reviewer agent, and the routing script all run as GitHub Actions jobs triggered by the PR — never “someone remembers to invoke it.” A red reviewer verdict blocks a merge exactly like a failing test.

  • Required-check results (the 11 dots)
  • Branch-protection rules themselves
  • Review approvals & requested reviewers
  • Routing labels (auto-merge / needs-human)
In the Console
  • Nothing durable. It holds no state.
  • It reads the two columns to the left…
  • …and writes only via the CLI + PR flow.
  • Restart it, move machines, close the tab —
  • you lose zero information.

This is the “statelessness doctrine,” and it isn't aspirational — it's covered by negative guarantee tests that fail the build if the server ever writes state anywhere but git, the GitHub API, or the approval-PR flow.

Example: follow one click — what “Approve Spec” really does Example

You click one friendly button. Behind the glass it is exactly this sequence of ordinary git + GitHub operations — nothing you couldn't type yourself (and the ⌘ popover shows you the commands):

what the Console shells out to do
# 1. put the drafted spec delta on a branch
git checkout -b spec/OMS-142-partial-cancellation
git add openspec/changes/partial-cancellation
git commit -m "spec(OMS-142): partial cancellation"

# 2. open a PR to the CODEOWNERS-protected specs path
gh pr create --base main --fill

# 3. CODEOWNERS requires YOU on this path — you approve.
#    that approval is itself a durable git object, not a UI flag.
gh pr review --approve  &&  gh pr merge --squash

After the merge, nothing “told” the system the state changed. The next time anything reads workorder.yaml plus the merged commit, the feature simply is in SPEC_APPROVED. State is derived from git, not stored beside it.

and because it's all git, the audit trail is free
git log --oneline -- workorders/OMS-142-partial-cancellation/workorder.yaml
a1c9f22 OMS-142: ORACLES_APPROVED   # ← your oracle approval, as a commit
7e0b418 OMS-142: SPEC_APPROVED      # ← the click above
3d5aa90 OMS-142: DRAFT_SPEC         # ← crucible new
Why bother being stateless? Isn't a normal app database simpler? Why this?

A separate database would be simpler to build — and would quietly undo the whole point:

  • No second source of truth to drift. If approvals lived in an app DB, a bug or an outage could disagree with git about what was approved. With one source, disagreement is impossible.
  • The fallback is total. Because the app owns nothing, “the app is down” is never “work is blocked.” Every action has a typed equivalent, so a crash costs ergonomics, not progress.
  • Tamper-evidence for free. An approval that is a signed git commit can't be silently flipped in a database row. The history is the audit log.
  • The guarantee doesn't depend on the app being correct. Enforcement lives in CI and branch protection, so even a buggy Console cannot let unverified code through.

06 — The decisions & the opinions

Why it's shaped this way

Crucible is deliberately opinionated. Here are the load-bearing choices and what was rejected.

ELI5

The biggest opinion in one line: the rules don't live in the app.

Imagine a diet app. If the only thing stopping you from eating cake is a friendly reminder, you'll eat the cake on a bad day. Crucible instead puts the locks in git and GitHub themselves — the deep plumbing everything runs on. So the same walls stand even for the boss, even at 2 a.m., even when you're in a hurry. There is no “skip” button to find.

The human's role, steady-state

Once calibrated, the owner's work is exclusively judgment work:

  • Author/approve specs, with an agent as drafting partner that surfaces edge cases.
  • Author or line-by-line review oracles — the real “code review,” done up front on the highest-leverage artifact.
  • Review risk-routed PRs (~15–20%): auth, money movement, data deletion/migration, public API surface, dependency additions, concurrency — plus anything the reviewer agent flags.
  • Random 10% sample audits of auto-merged PRs — statistical process control on the pipeline itself.
  • Resolve escalations — always as a spec/oracle fix.
  • Improve the harness — rubric lines, architecture rules, skills, prompts — driven by the ratchet.

Line-by-line human review is mandatory for exactly one category with no exceptions: the harness itself and the Gauntlet definitions — the trusted computing base of everything downstream.

07 — How to use Crucible

From an empty repo to shipping software

First, a one-time setup that turns a fresh repo into a Crucible-governed project and gets the Console running. Then the same short loop, every feature, forever.

Part 1 — One-time project setup

You do this once per software system. It scaffolds the governed structure into a new repo, wires up enforcement, and gets you a running Console pointed at the project.

ELI5

Setting up is like installing the referees and the rulebook before the game starts. You run one command (crucible init) and it drops in all the rules, the automatic judges, and the “only the owner can change the rules” locks. Then you turn on the dashboard (the Console) and point it at your project. After that, you never set up again — you just build features.

What you need first — the prerequisites

Crucible reuses tools you very likely already have. Nothing here is bespoke; it deliberately rides on git, GitHub, Docker, and the CLIs you already log into.

git + a git repo

The project must be a git repository — init refuses otherwise. Run git init (or clone an empty repo) first. Git is the database, so this is non-negotiable.

gh — GitHub CLI, logged in

Authenticated via gh auth login. The Console reuses your existing gh auth token — no separate PAT needed. Used to open PRs, apply branch protection, and set the reviewer secret.

claude — Claude Code, logged in

The spec/oracle chat and the implementer run on Claude. The Console reuses your logged-in host claude; if it's set up, you need no token in .env for local use.

Node.js ≥ 22

Runs the @crucible/cli (the crucible binary) and the Console server + web app. The CLI declares engines: node >=22. Node 21 still runs, but prints a harmless ExperimentalWarning: Importing JSON modules — match 22 to silence it. If you use nvm, note that the crucible, openspec and node binaries all live in the active version's bin, so run every step in a shell where the right version is selected (nvm use 22).

openspec — the SDD engine

Install globally with npm i -g @fission-ai/openspec. Required before crucible init — init shells out to openspec init to lay down the spec-driven structure. If openspec isn't on PATH, init fails (see the troubleshooting note below).

Docker

The sandbox runner (crucible run) executes the agent inside the pinned toolchain image with the network off. Also lets you run the Console as a localhost container if you prefer.

A GitHub repo you own

Push access and permission to set branch protection — that platform-level wall is what makes the required checks unbypassable. (The java profile also wants a JDK to generate the Gradle wrapper.)

Check yourself in ten seconds git --version · gh auth status · claude --version · node --version · openspec --version · docker --version — if all six answer (and node is ≥ 22), you're ready to init.

Initialize the project

  1. Create (or enter) the consumer repo · You

    Each software system lives in its own repo (that's Model B). Start it as a git repo.

    Terminal: mkdir order-system && cd order-system && git init

  2. Install openspec · You

    crucible init drives openspec under the hood, so it must be installed and on PATH first.

    Terminal: npm i -g @fission-ai/openspec

  3. Install the CLI · You

    The crucible binary is not yet published to npm. Today you install it from the framework repo with npm link, which builds it and puts crucible on your PATH (as a symlink to the local build, so rebuilds pick up automatically).

    Terminal: cd path/to/crucible/packages/cli && npm ci && npm run build && npm link Verify: which crucible && crucible --version # resolves under the active node's bin

    nvm note: the link lives in the active node version's bin. If you switch versions with nvm use, re-run npm link under the new one. (Once published, this becomes a plain npm i -g @crucible/cli.)

  4. Run crucible init · You

    One command turns the repo into a governed project: it runs openspec init, installs the oracle-driven schema, and emits CODEOWNERS, the Gauntlet CI wrapper, the branch-protection ruleset, CLAUDE.md, the Crucible skills, the directory skeleton, and crucible.yaml. It never overwrites existing files.

    Terminal: crucible init --owner your-gh-handle # --lang java (default)

  5. Finish the wiring · You

    init prints the exact next steps. In order:

    1. gradle wrapper --gradle-version 9.6.1 # java profile only — see the bootstrap note below 2. git add -A && git commit -m "chore: crucible init" && git push -u origin main 3. settings/apply.sh # GitHub governance (wall + labels + perms) — see the plan note below 4. claude setup-token # mint a Claude token, once 5. gh secret set CLAUDE_CODE_OAUTH_TOKEN # for the reviewer CI job

    Put that same token in a local .env as CLAUDE_CODE_OAUTH_TOKEN=… so crucible run can drive the sandbox on your machine. If you drive everything through the Console with a logged-in host claude, steps 4–5 are only needed for the CI reviewer job (GitHub Actions) — they aren't used locally. But note: once branch protection is on, review / reviewer-verdict is a required check, so without this secret no PR can merge through the wall. Local drafting works; the merge tail does not. Set the secret before you expect a PR to land.

Bootstrap note — the Gradle wrapper (step 1) Step 1 assumes a gradle binary is already installed, which is a chicken-and-egg: a fresh machine has only a JDK. Get one wrapper, once, either way — then it's committed and every later build uses ./gradlew: a) install Gradle: brew install gradle # or sdkman: sdk install gradle 9.6.1 b) or no install — generate from the distribution: curl -L -o /tmp/g.zip https://services.gradle.org/distributions/gradle-9.6.1-bin.zip && unzip -q /tmp/g.zip -d /tmp mkdir /tmp/wg && (cd /tmp/wg && echo 'rootProject.name="w"' > settings.gradle.kts && /tmp/gradle-9.6.1/bin/gradle wrapper --gradle-version 9.6.1) cp -r /tmp/wg/gradlew /tmp/wg/gradlew.bat /tmp/wg/gradle ./ # into your repo; needs JDK 17+ Generate in an empty dir with a one-line settings.gradle.kts so Gradle doesn't try to resolve your project's plugins. (Gradle 9 refuses to run any task, even wrapper, without a settings file.)
Add a .gitignore before you commit The java profile does not currently emit one, so the first ./gradlew build leaves .gradle/ and build/ as untracked clutter. Add a minimal ignore first: printf '.gradle/\nbuild/\n**/build/\n.env\n' > .gitignore
Plan note — branch protection needs a public repo or a paid plan (step 3) settings/apply.sh installs a ruleset for required checks — the enforcement wall. GitHub only allows rulesets/branch protection on a private repo with Pro / Team / Enterprise; a private free repo returns 403 "Upgrade to GitHub Pro or make this repository public". Options: (a) make the repo public — rulesets are free there; (b) upgrade the plan; (c) proceed without the wall for now — the Gauntlet CI still runs on every PR and the whole Console flow works, but merges aren't hard-blocked, so it's not tamper-proof until protection is on. The script's other actions (labels, workflow-token = write, auto-merge) work on any plan; because the script is set -e, a ruleset 403 aborts it before those run, so apply them separately or re-run after enabling protection.
Troubleshooting: init fails with a blank environment error: openspec init failed: Gotcha

The message after the colon is empty because init only surfaces the child command's stderr, and openspec prints to stdout. The blank error almost always means openspec could not be found on the PATH of the shell that launched crucible — typically because a different (or default) node version is active and the global openspec/crucible links live under a different nvm version's bin.

Fix: run everything in one shell with the right toolchain resolved:

1. nvm use 22 # the version you linked crucible + installed openspec under 2. which crucible openspec # both must resolve before you continue 3. crucible init --owner your-gh-handle

Sanity check that openspec itself works in that repo: openspec init --tools claude --force should exit 0. If it does but crucible init doesn't, it's a PATH/version mismatch, not openspec.

Why reuse my host gh and claude instead of asking for tokens? Why this?

The Console is meant to be plug-and-play on your own machine, and to hold as little as possible:

  • No new secret to manage. If you can already gh and claude from your terminal, the Console just borrows those sessions (gh auth token, your logged-in claude). Fewer copies of a token is fewer ways to leak one.
  • Localhost only. The server binds to 127.0.0.1 — it's a personal instrument, not a shared service, so reusing your local identity is exactly right.
  • Explicit override still exists. Need a different identity, a service token, or a different port? Drop a console/.env and it takes precedence. The default is just the friction-free path.

Part 2 — The everyday loop, step by step

One feature, end to end. Roughly 60–90 minutes of human time — almost all of it in the spec and oracles. Everything after launch is largely unattended.

ELI5

Your whole day is four things:

  • 1. Say what you want. Chat with the AI until the feature is written down as clear, exact rules.
  • 2. Approve the tests. Look at the automatic judges the AI drafted and say yes. This is the one step you cannot skip.
  • 3. Press start and walk away. The AI writes the code; the machines check it.
  • 4. Only if it's risky, do a 15-minute read. Most changes you never have to look at.

That's it. You make the important calls; the machines do the grinding.

  1. Create the work order · You · ~2 min

    Open the Console → New Feature. Fill in an ID, title, and change slug. Click Create. The work-order file is scaffolded.

    Terminal: crucible new OMS-142 --title "Partial cancellation" --change partial-cancellation State: DRAFT_SPEC

  2. Draft the spec conversationally · You + Agent · 20–40 min

    In the spec chat panel, start the change. The agent drafts the proposal and spec deltas, and probes edge cases — “when the last item is cancelled, does the order become CANCELLED?     When satisfied, click Approve Spec — the Console opens a PR to the protected path; CODEOWNERS requires you; you approve; it merges. Your approval is now a durable git commit.

    State: SPEC_APPROVED

  3. Draft & approve the oracles · You + Agent · 15–30 min

    The agent drafts the traceability table (oracles.md) and drafts oracle implementations on a branch — property tests, concurrency tests, DB constraints — one or more per requirement.

    Open Oracle Review. This is your real code review, done up front. Any unmapped requirement shows red at the top. Read each oracle's source line by line and ask: would this fail if the requirement were violated in the ways I care about? Then click Approve Oracles.

    Mandatory gateYour approval of the oracles is required before any code is written. This is the guarantee against agent-authored vacuous verification.
    State: ORACLES_APPROVED

  4. Finish planning artifacts · You · ~5 min

    Continue until design.md and tasks.md exist (tasks unblocks now that oracles are done). Skim both — you're checking scope sanity, not wording. tasks.md is the agent's own checklist.

    Open Run Monitor. Click Validate — the precondition report renders inline (schema valid, artifacts done, oracles merged, module map non-empty). Then Package, then Start Implementation.

    Terminal: crucible validate OMS-142 --advance && crucible package OMS-142 && crucible run OMS-142 State: PACKAGED → IMPLEMENTING

  6. Walk away · Auto + Agent · 0 min

    A fresh sandbox spins up from the pinned image, no network, bundle read-only. The agent works through tasks.md, writes its own tests underneath your oracles, and iterates. It cannot touch /oracles, /specs, or /ci. On green, the runner (not the agent) opens the PR.

    The Gauntlet runs all eleven gates. If mutation testing fails at 71%, that output goes back as input to a new attempt; the agent strengthens its tests; the next push is green. The reviewer agent (separate context) posts its verdict.

    State: PR_OPEN → GATES_GREEN → AI_REVIEWED

  7. Your 15-minute review — only if routed to you · You

    Routing is deterministic: if the PR touches a risk path (e.g. refund/**) or carries a reviewer flag, it comes to you. Otherwise it auto-merges. In the Review Queue you get three panes: the diff, the reviewer's verdict table, and the spec. The gates already proved behavior — you read only for what machines can't judge: is the transaction boundary sensible, is this a design you want to live with? Approve.

    State: ROUTED → MERGED → CANARY → DONE

  8. Archive · You · ~1 min

    In chat: /opsx:archive. The guard confirms all PRs are merged, then merges the spec deltas into the living specs.

    State: ARCHIVED

What you did NOT do: read the implementation while it was written, babysit the sandbox, or manually check any gate. What you DID do: made every judgment call — requirements, oracle quality, the one risky design decision.

The fallbacks — nothing is trapped in the app

The Console is a view and remote control, never the state machine itself. All authoritative state lives in git + GitHub. This isn't a nice-to-have; it's a tested acceptance criterion.

ELI5

If the nice app crashes, you lose nothing. Every button in it is just a shortcut for a command you can type yourself. And everything it shows you is read straight from git and GitHub, which you can open on your own.

It's like a fancy TV remote: if the remote breaks, the TV still has buttons on it. The app is the remote. Git is the TV. That's what “git is the database” means — the real stuff lives in git, not in the app.

Console down / not yet built

Run the identical state machine through its native interfaces: crucible new, then Claude Code in the repo for the spec/oracle chat, approve via gh pr create + CODEOWNER merge, then crucible validate / package / run. The Gauntlet and reviewer never depended on the Console. The GitHub PR page already shows the reviewer verdict and the same eleven check dots.

The agent gets stuck (escalation)

When the spec is ambiguous or contradictory, the agent stops — it does not thrash to its budget — and writes escalation.md with the blocking requirements and a set of trade-off options. A notification fires. You decide, then fix it as a spec/oracle change, never as a code workaround. Re-validate, re-run. Ambiguity is a spec bug; the fix is always upstream.

A defect escapes to production

The hotfix goes through the loop, never around it: a small work order, oracle-first. The crucial move — confirm the new oracle fails against current main before the fix. A fix oracle that doesn't fail first proves nothing. Then the agent's fix is judged by a test that demonstrably catches the bug.

The ratchet closes the loop

Every postmortem's mandatory closing field is “Ratchet commit.” Ask: should a gate or the rubric have caught this? If so, add the rubric line or rule — a one-line PR. From that commit forward, the reviewer asks it on every relevant diff, forever. The system is now stricter than before the bug existed.

Example: what an escalation looks like Example

The agent discovers two approved rules that can't both be true. Instead of guessing — or thrashing until its iteration budget runs out — it writes one file and stops. The file lays out the conflict and the real options, but makes no decision:

/workorders/OMS-155-auto-release-holds/escalation.md
# Escalation — OMS-155
Attempt 3 of 6. Run halted.

Blocking: REQ-AR-2  vs  ORA-SETTLE-1
Conflict: REQ-AR-2 says release the payment hold within 5 minutes
  of fulfillment, but ORA-SETTLE-1 requires ALL ledger mutations to
  happen inside the nightly settlement batch. Both cannot hold.

Options:
  A) Relax REQ-AR-2 to "by next settlement batch"
  B) Add a real-time ledger path with a new invariant
  C) Release the hold immediately, post the ledger entry at settlement

A notification fires. The human picks (say) option C — a genuine product decision that must never be agent improvisation — and resolves it upstream, as a spec/oracle change, then re-runs against an unambiguous spec. The escalation file, the decision, and the amended spec all become permanent git history: six months later you'll know exactly why holds release immediately but settle nightly.

08 — FAQ

Frequently asked questions

The questions that come up on first contact with the model.

Q Isn't this just TDD?

It borrows TDD's mechanism (checks before code) but not its granularity. It's closer to Design by Contract + BDD: business rules and system-level properties, authored by a human, holding across all features. The agent is still free to do classic red-green-refactor TDD in its own inner loop underneath the oracles.

Q If the AI writes the code, who writes the tests?

Two layers. The oracles — the authoritative tests that judge the requirement — are authored or line-by-line reviewed and approved by a human, before any code exists, and are unmodifiable by agents. The agent then writes its own additional unit tests underneath those oracles as it implements. Mutation testing keeps the agent's tests honest.

Q What stops an agent from just gaming the checks?

Several layers stacked deliberately. Oracles it can't touch. Mutation testing that breaks the code and demands a test scream (catching weakened assertions and vacuous tests). A hermetic, network-off, fresh-per-run sandbox that closes degenerate paths like installing a different library or mutating shared state. A work-order legitimacy check that rejects any write outside the declared module map. And an adversarial reviewer with a separate context. Gaming one gate still leaves the others.

Q Why put enforcement in git/CI instead of the app?

Because enforcement has to survive someone bypassing the tooling — including the owner, tempted by a quick fix on a bad day. If the only guardrail is a convenient app, the first shortcut erodes the system. So the app is convenience and branch protection + CODEOWNERS + required checks are legitimacy. Delete the app and the workflow is still enforced — just more tedious.

Q Does a human still read any code?

Yes — but as a precision instrument, not a coverage instrument. Line-by-line review is kept for a mechanically-identified high-risk slice (~15–20%: auth, money, data deletion, public APIs, dependency additions, concurrency), plus anything the reviewer agent flags, plus a random 10% audit sample. And one category is always reviewed line-by-line: the harness and the Gauntlet definitions themselves — the trusted computing base.

Q What happens when the spec is ambiguous or contradictory?

The agent escalates — it writes a structured escalation.md with the conflict and a set of trade-off options, and its run ends. It does not improvise. A human makes the product decision and resolves it as a spec/oracle change. Ambiguity is treated as a spec bug, and the fix is always upstream, never a downstream workaround.

Q Can I do a quick hotfix outside the process?

No — and that's the point. Hotfixes go through the loop too, oracle-first, with the extra discipline of proving the new oracle fails on current main before the fix. The loop is deliberately fast enough that “going around it” would save minutes and cost the guarantee. There is no privileged path, not even for the owner.

Q Why Java?

Two reasons. It's the owner's area of expertise, which matters because the human's remaining review is high-consequence judgment work. And it has a best-in-class machine-checkable ecosystem — ArchUnit for architecture, PIT for mutation testing, Error Prone for strict typing, jqwik for property tests — which turns hallucinated interfaces into compile errors and makes the gates sharp. The framework's concepts aren't Java-specific, but the reference target is.

Q What if the Console breaks mid-feature?

You lose nothing. Every Console action has a terminal equivalent, and every fact it displays is read from git + GitHub, which you can open directly. The “kill the Console mid-workflow and complete every operation from terminal + GitHub UI with no data loss” drill is an explicit acceptance criterion. Git is the database; there is no second source of truth to lose.

Q Why not use a workflow engine like Temporal or an agent swarm?

Premature complexity for solo/small-team scale. Git + CI + a thin CLI is deterministic, auditable via git history, and debuggable with cat and git log. The whole design keeps model calls out of the control flow — routing and orchestration are plain scripts — precisely so the loop is reproducible. A mega-prompt that self-enforces the workflow was rejected for the same reason: LLMs don't reliably self-enforce multi-step processes.

Q How does it avoid getting worse over time?

The ratchet. Every escaped defect's postmortem must produce a new oracle, rubric line, or architecture rule — as a commit, not a note. The system only ever gets stricter. And the harness itself has a pipeline eval suite — benchmark tasks re-run on any model, prompt, or skill change — so quality regressions in the tooling are caught before they ship.

Q Do I turn on auto-merge from day one?

No. There's a calibration period — roughly two weeks — where the owner reviews everything, so the rubric, risk list, and diff caps get tuned against the agent's actual failure modes rather than hypothetical ones. Trust in the gates is earned by watching them catch things you would have caught, and only then do you mechanically retreat to sampling.

09 — Mini-dictionary

The words, decoded

Every unfamiliar or jargon-y term on this page, with its plain dictionary meaning and what it specifically means inside Crucible.

ArchUnit proper noun

DictionaryA Java library for writing tests that assert facts about a codebase's architecture (which package may depend on which, etc.).

In CrucibleThe architecture gate. It encodes rules like “only OrderStateMachine may change an order's status” or “the web layer may not import the database layer,” so that structure can't quietly erode across hundreds of agent PRs.

Auto-rollback noun

DictionaryAutomatically undoing a software release when health signals turn bad, without waiting for a human.

In CrucibleThe Layer-4 safety net: if a canary breaches its SLO, the deploy is reverted automatically, and the incident feeds the ratchet.

Branch protection / required status check noun

DictionaryGitHub settings that forbid merging to a branch until named checks pass and rules (like required reviewers) are satisfied.

In CrucibleWhere enforcement actually lives — not in the app. Nobody, the owner included, can merge around the Gauntlet, because the platform itself refuses.

Canary deploy noun

DictionaryReleasing a change to a small slice of traffic first — the “canary in the coal mine” — and watching it before a full rollout.

In CrucibleThe first runtime (Layer 4) check after merge: the change serves a little real traffic under SLO monitoring before it reaches everyone.

CODEOWNERS noun

DictionaryA GitHub file that maps file paths to the people whose approval is required to change those paths.

In CrucibleThe mechanism that makes /specs, /oracles, and /harness human-approval-only — and literally unwritable from an agent's implementation branch.

Crucible noun

DictionaryA heat-resistant vessel in which metals or other substances are melted and purified at very high temperatures.

In CrucibleThe whole system: a container of enforcement that never yields, inside which agents do fast, high-volume work — and out of which nothing passes unverified.

CVE noun

Dictionary“Common Vulnerabilities and Exposures” — an entry in a public catalog of a known, identified security vulnerability.

In CrucibleWhat the dependency-scan gate checks for: a PR that pulls in a package with a known CVE (or a risky license) cannot go green.

Deterministic adjective

DictionaryProducing the same output from the same input every time, with no randomness or hidden variation.

In CrucibleThe property the whole control flow is designed for. Routing and orchestration make no model calls, so the same work order always moves the same way — reproducible and debuggable with cat and git log.

Ephemeral adjective

DictionaryLasting for a very short time; transitory.

In CrucibleThe sandbox is created fresh for a single run and destroyed afterward, so nothing — no cached state, no leftover file — carries between runs.

Escalation noun

DictionaryReferring a matter to a higher authority for a decision.

In CrucibleA structured file (escalation.md) an agent writes when the spec is ambiguous or self-contradictory. It ends the run and hands the decision to a human — the agent never improvises past ambiguity.

The Gauntlet noun

DictionaryHistorically, an ordeal in which someone passes between two rows of people who strike them; figuratively, a demanding series of trials to get through.

In CrucibleThe full set of required CI checks — eleven gates — that every PR must survive to merge. Together they replace routine line-by-line human review.

Harness noun

DictionaryA set of straps and fittings used to control or direct an animal or a person.

In CrucibleEverything around the model that shapes its behavior: CLAUDE.md, skills, prompt templates, the sandbox runner, permission config, the CLI. Treated as a product — versioned, reviewed, regression-tested.

Hermetic adjective

DictionarySealed so completely that no outside air or influence can get in; airtight.

In CrucibleA build or sandbox that uses only declared, pinned inputs — no network, no ambient machine state — so results can't drift, and the code under test can't reach outside to cheat its way to green.

Idempotent adjective

DictionaryOf an operation: producing the same result whether it's applied once or many times.

In CrucibleA common requirement worth an oracle — e.g., firing the same cancellation request five times must still yield exactly one refund. A property/concurrency test proves it.

Mutation testing noun

DictionaryA technique that deliberately introduces small faults (“mutants”) into code to measure how well a test suite detects them.

In CrucibleA required gate (PIT). Surviving mutants prove the tests are vacuous or the assertions weakened — the main structural defense against an agent reward-hacking its way to green.

Oracle noun

DictionaryA source believed to give wise, authoritative answers or judgments.

In CrucibleA machine-executable judge of one requirement — a property test, contract test, DB constraint, or rule — that decides pass/fail with no human in the loop. “A requirement without an oracle is a wish.”

Orchestrator noun

DictionarySomething that coordinates many separate parts into a single coherent whole.

In CrucibleDeliberately not a smart engine: just git + CI + a thin CLI. The “loop” is plain deterministic scripts, so it can be inspected and replayed rather than trusted.

Property test noun

DictionaryA test that asserts some property holds across many automatically-generated inputs, instead of checking one fixed example.

In CrucibleThe preferred shape for an oracle — “for any order, refunds never exceed the amount captured” — because it catches whole classes of bug that a single example (“a $10 item”) would miss.

Ratchet noun

DictionaryA device with angled teeth that allows motion in only one direction and prevents it from slipping back.

In CrucibleThe rule that every escaped defect must produce a new oracle, rubric line, or architecture rule — as a commit. Strictness only ever increases; the system can't slip backward.

Reward hacking noun

Dictionary(Machine learning) An agent satisfying the literal letter of its objective in an unintended, degenerate way that misses the point.

In CrucibleThe behavior most gates are pointed at: weakening an assertion, mocking away a failing component, or pinning a different library “to make the check pass.” Hermetic sandboxes and mutation testing close these paths.

SAST noun

Dictionary“Static Application Security Testing” — analyzing source code for security flaws without running it.

In CrucibleA Gauntlet gate (Semgrep). It catches injection, path traversal, hardcoded secrets, and weak crypto — vulnerabilities with no functional symptom that behavioral tests are blind to.

SDD / OpenSpec noun

DictionarySDD = “Spec-Driven Development,” a workflow where written specifications drive the implementation. OpenSpec is one such tool.

In CrucibleOpenSpec is the SDD mechanism Crucible builds on. Crucible forks its schema to insert oracles as a mandatory artifact. Slogan: “OpenSpec provides direction; Crucible's CI provides legitimacy.”

Slopsquatting noun

Dictionary(Neologism) Registering malicious packages under names that AI models tend to hallucinate, so that an AI “imports” one by mistake.

In CrucibleA specific supply-chain risk the dependency allowlist + scan gate exists to stop: agents can't pull in a package that isn't explicitly allowlisted.

SLO noun

Dictionary“Service Level Objective” — a target for a reliability metric, e.g. “99.9% of requests complete under 200 ms.”

In CrucibleThe thresholds a canary is judged against. A breach is what triggers auto-rollback at Layer 4.

State machine noun

DictionaryA model in which something is always in exactly one of a finite set of states and moves between them only along defined transitions.

In CrucibleHow every feature is modeled: a work order moves DRAFT_SPEC → … → ARCHIVED, and every transition has a gatekeeper that is not the agent.

Traceability noun

DictionaryThe ability to follow the links of something through each step of a process.

In CrucibleThe mapping from every SHALL/MUST requirement to the oracle(s) that judge it. A linter fails the build if any requirement is left unmapped — enforcing “a requirement without an oracle is a wish.”

Trusted computing base noun

Dictionary(Security) The set of components that must be correct for a system's guarantees to hold at all.

In CrucibleThe harness and the Gauntlet definitions themselves — the one category of code a human reviews line-by-line, always, because everything downstream trusts it.

Vacuous verification noun

DictionaryChecking that looks rigorous but actually confirms nothing meaningful.

In CrucibleAgent-written tests that describe what the code does rather than what it should do — forty green tests encoding one misunderstanding. Caught by human-owned oracles plus mutation testing.

Crucible — a container of enforcement that never yields.
Humans author the judges; agents author the code; git guarantees the rest.
